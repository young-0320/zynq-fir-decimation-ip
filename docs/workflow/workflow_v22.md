## 0. 쉬운 설명 — 공장 라인 비유 (버그가 처음이면 여기부터)

이 래퍼를 공장 라인 하나로 그리면 4개 버그가 같은 그림 위에서 보인다.

- **투입구**: 샘플이 한 개씩 들어간다.
- **긴 처리 터널**: 들어간 샘플이 완성되어 나오기까지 여러 칸을 거친다(세차장 터널).
  v1=4칸, v2=5칸. 즉 지금 넣어도 결과는 4~5칸 뒤에 나온다.
- **출구 회전문**: 완성품 2개 중 1개만 통과시킨다(M=2 데시메이션). "통과/막기"를 번갈아 한다.
- **픽업 선반(3칸)**: 완성품을 잠깐 올려두는 곳 = skid buffer.
- **가져가는 트럭**: 하류(DMA). 바쁘면 "잠깐 멈춰" 신호를 준다 = backpressure(`tready=0`).
  그동안 선반에 쌓인다.
- **끝 딱지(TLAST)**: 한 주문(패킷)의 마지막 완성품에 붙여야 트럭이 "이 주문 끝"을 안다.

**핵심 한 문장**: _투입구를 막아도, 터널 안에 이미 들어간 것들은 계속 나온다._
이 당연한 사실을 아래 4곳이 각각 깜빡했다.

- **버그 1 (선반 넘침)**: "선반 2번 칸 차면 그만 받기"로 막는데, 막은 뒤에도 터널에서
  완성품이 더 나와 3칸을 넘치면 조용히 드랍. 그게 끝 딱지면 트럭 무한 대기.
- **버그 2 (무한 방출)**: 주문 종료 판정이 "센 개수 == 목표"라, backpressure로 개수가
  목표를 지나치면 영원히 안 끝나고 0을 무한 토출.
- **버그 3 (찌꺼기 넘김, 제일 시급)**: 터널 비우려고 뒤에 0을 과다 주입해서, 여분이 다음
  주문 앞에 낀다. 펌웨어가 주문 사이 리셋을 안 해서 누적됨. backpressure 없이도 발생.
- **버그 4 (회전문 어긋남)**: 홀수 길이 개수 공식이 틀리고, 회전문 "통과/막기" 차례가
  주문 사이에 어긋남(버그 3과 한 뿌리).

> 재현 수치, 실보드가 왜 통과했는지의 해석, 버퍼 깊이 실측 등 **모든 결론은
> `docs/log/41_axis_skid_buffer_bug_sim.md`**. 이 문서는 그 결론을 바탕으로 한 **계획**만 담는다.

## 배경

workflow_v21(v2 파이프라인 분할 + clk_wiz 정밀 스윕)을 마친 뒤, 다음 단계(v1@115MHz /
v2@145MHz 보드 실측)로 넘어가기 전에 별도 검토를 진행하였다.
검토 대상은 이전에 코드 리뷰에서 "미확인 코너케이스"로 남겨뒀던 AXIS 래퍼의
3단 skid buffer였다(`docs/log/40_clkwiz_precision_sweep_v1_v2_fmax.md` 이후 세션에서
직접 짚었던 질문).

검토 결과 **skid buffer 드랍 의심이 사실로 확인됐고, 추가로 3개 버그가 더 발견됐다.**
재현 테스트벤치(`tb_skid_stress.sv`, `tb_buf_depth.sv` — repo 편입 완료, `make run_bug`)로
iverilog 시뮬레이션까지 실행해 확인한 결과이며, 추측이 아니다. 이후 본인이 코드를 다시 읽고
Verilog 의미론(같은 always 블록 내 non-blocking 대입의 old-value 참조)으로 핵심 레이스를
독립적으로 재검증했고, 실제 배포된 C 펌웨어(`sw/fir_decimator_demo.c`)를 확인해 심각도를
재평가했다.

**가장 중요한 확인 사항**: `sw/fir_decimator_demo.c`의 `main()`은 `while(1)` 루프 안에서
UART 명령을 받을 때마다 `dma_run()`을 반복 호출한다. `dma_run()`은 **AXI DMA IP만
soft-reset**하고(`MM2S_DMACR` bit 2), PL 쪽 커스텀 래퍼(`waiting_for_last_out`,
`out_cnt`, `keep_next` 등)는 리셋하지 않는다. 즉 **한 번 부팅한 세션에서 UART 명령을
2번 이상 보내면(현재 데모의 정상적인 사용 패턴 그 자체) 래퍼 내부 상태가 패킷 간에
누적/오염될 수 있다.** 아래 버그 3은 backpressure 없이도, 정상적인 반복 사용만으로
발생한다 — 즉 이론적 코너케이스가 아니라 **현재 데모 워크플로 자체가 매번 밟고 있을
가능성이 있는 경로**다.

## 발견된 버그 4개

공통: `fir_decimator_n43_axis.v`(v1)과 `fir_decimator_n43_axis_v2.v`(v2) 양쪽 다
해당(v2가 새로 만든 버그 아님 — v1부터 있던 로직을 그대로 재사용했기 때문).

### 버그 1. Skid buffer 드랍 (3-slot 용량 부족)

**증상**: 연속 입력(버블 없음) + 다운스트림 `m_axis_tready` 장기 low 조합에서, 코어가
새 출력을 내는 순간 버퍼 3칸이 이미 다 차 있으면 그 샘플이 조용히 사라진다. 하필 TLAST
비트가 드랍되면 S2MM이 무한 대기. (재현·수치 → log 41)

**근본 원인**: `core_ready = ~valid1` 게이팅은 버퍼 점유 2 이상일 때 새 입력을 막지만,
그 시점에 이미 "accepted 됐지만 아직 core_out_valid로 안 나온" 샘플이 파이프라인 안에
남아있다. 코어의 실제 accept-to-output latency는 FIR 단독이 아니라
`decimator_m2_phase0.v`의 +1 레지스터를 포함해 **v1=4 cycle, v2=5 cycle**이다.
3-slot 버퍼는 이 latency를 감당하기에 부족하다(설계 당시 FIR 3-cycle 기준으로만
사이징된 것으로 추정).

**핵심**: 새 입력 차단(게이트)과 파이프라인 잔여 드레인은 별개로 동작하므로, 게이트가
닫힌 뒤에도 in-flight 출력이 버퍼로 계속 들어온다 — 이게 3칸 부족의 원인. v1/v2 필요
깊이가 같은 이유(데시메이션 패리티가 v2의 +1 latency 흡수)와 사이클 단위 유도는 log 41.

### 버그 2. 무한 flush 데드락 (버그 1보다 심각)

**증상**: `out_cnt + 1 == target_out_cnt` 정확 등호 비교이기 때문에, backpressure로
인해 (등호가 성립하는) 정확한 사이클을 놓치면 `waiting_for_last_out`이 영원히 안 풀리고
flush가 0을 무한 주입한다. (재현·수치 → log 41)

**근본 원인**: 주 발동 경로는 "같은 사이클 우연"이 아니라 **다사이클 윈도우**다 —
backpressure로 마지막 입력(tlast)의 수락이 지연되는 동안, 파이프라인에 이미 들어가 있던
샘플은 backpressure와 무관하게 계속 드레인되므로(버그 1과 동일 메커니즘) `out_cnt`가
`target_out_cnt` 세팅보다 **여러 사이클 먼저** 그 경계값을 지나쳐 버린다. 이후
`waiting_for_last_out`이 1이 됐을 때는 이미 `out_cnt`가 그 값을 통과해 등호가 다시는
성립하지 않는다. (부차적으로, tlast를 인식하는 바로 그 사이클에는 같은 `always` 블록의
종료조건이 `waiting_for_last_out`의 old value(0)로 평가되는 non-blocking 특성도 겹친다 —
이건 하나의 하위 케이스일 뿐 주 원인은 위의 다사이클 드레인이다.)

**⚠ `== → >=` 단순 치환으로는 부족(분류 재조정)**: 이 버그를 "저위험 국소 수정"으로
보면 안 된다. 두 겹이 더 있다.

- **(a) 종료 판정이 `core_out_valid` 펄스에 게이트되어 있다.** flush 주입을 없애면(아래
  버그 3 수정 방향) `out_cnt`가 target을 지난 뒤에는 판정을 평가할 `core_out_valid`
  펄스 자체가 다시 오지 않는다 — `>=`로 바꿔도 레벨 기반 판정으로 옮기지 않으면 여전히
  잠긴다.
- **(b) TLAST가 이미 유실된 상태다.** `out_cnt`가 target을 넘어섰다는 건 TLAST를 달았어야
  할 비트가 이미 `tlast0=0`으로 버퍼를 통과했다는 뜻이다. FSM을 `>=`로 회복시켜도 그
  패킷의 `m_axis_tlast`는 복구되지 않는다. TLAST 태깅 시점(코어 출력 시점)이 "target이
  아직 미확정일 수 있는" 레이스를 안고 있어서, **버그 2는 사실상 버그 3·4와 같은 설계
  레벨 수정 묶음**이다(근본 원인 요약 표 참고).

> 실보드가 왜 이걸 안 밟았는지의 해석(잠복 조건)은 log 41 §3.

### 버그 3. 백프레셔 없이도 발생하는 패킷 경계 오염 — **가장 시급**

**증상**: `m_axis_tready=1` 고정(backpressure 전혀 없음)에서도 정상 출력 뒤에 stray
샘플이 붙고, 종료 후 `out_cnt` 잔류 + v1은 `keep_next` 위상까지 반전된다. (재현·수치 → log 41)

**근본 원인**: flush가 `waiting_for_last_out=1`인 동안 매 사이클 0-데이터를 계속
주입하는데, 이 주입은 "파이프라인을 드레인하는 데 필요한 만큼(고정된 L cycle)"이 아니라
"출력 개수가 target에 도달할 때까지"라는 **간접적인 카운트 기반 조건**으로만 멈춘다.
버그 2의 등호 레이스와 무관하게, 이 간접 조건 자체가 정확히 필요한 만큼만 주입한다는
보장이 없어서 여분의 zero-sample이 코어에 들어가고, 그 여분 샘플들의 출력이 몇 사이클
뒤 **다음 패킷 스트림에 새어 들어간다.**

**⚠ 실사용 영향(펌웨어 확인 결과, 버그 3을 최우선으로 올린 근거)**:
`sw/fir_decimator_demo.c`의 `main()`이 리셋 없이 `dma_run()`을 반복 호출하는 구조이므로
(`dma_run()`은 DMA IP만 soft-reset, 래퍼 상태는 안 건드림), **한 세션에서 UART 명령을
2번째 이상 보내는 순간마다 오염이 누적될 수 있다.** backpressure도 adversarial 패턴도
필요 없이 정상적인 반복 사용 자체가 트리거다. (실보드 통과 해석은 log 41 §3, 확정 실험은
아래 "확인 필요 1".)

### 버그 4. 홀수 길이 패킷 target 계산 오류 (버그 3과 결합됨)

**증상**: `target_out_cnt <= (in_cnt + 1) >> 1` 은 `floor(N/2)`인데, decimator는
`ceil(N/2)`개를 내보낸다(패킷이 "keep" 위상으로 시작한다고 가정할 때) → 홀수 길이에서
TLAST가 한 출력 일찍 나온다(재현 → log 41). 현재 `N_IN=8192`(짝수)라 이 자체는 지금
당장 안 걸리지만, 짝수/홀수와 무관하게 **버그 4는 버그 3(위상 드리프트)과 결합되어
있다** — 아래 참고.

**중요한 결합 관계**: `target = ceil(N/2)` 공식은 "패킷이 항상 keep 위상으로 시작한다"는
전제에서만 맞다. `decimator_m2_phase0.v`의 `keep_next`는 모듈 전역 리셋에서만
초기화되고, **패킷 단위 재동기화 포트가 없다.** 버그 3(flush 과다 주입)이 패킷마다
core_in_valid 총 펄스 수(실제 입력 + flush 주입)의 홀짝을 흔들면, 다음 패킷이 시작되는
위상 자체가 매번 달라질 수 있다 — 이러면 짝수 N이라도 target 공식이 맞다는 보장이
없어진다. **버그 3과 버그 4는 사실상 하나의 근본 원인(패킷 간 위상 비동기화)의 두
증상이다.**

## 근본 원인 요약

| 버그                        | 독립 수정 가능?            | 비고                                                                                                                       |
| --------------------------- | -------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 1 (skid 드랍)               | 예 — 버퍼 용량 확장       | 저위험, 시뮬레이션으로 직접 검증 가능. v1·v2 필요 깊이 동일할 가능성                                                      |
| 2 (무한 flush + TLAST 유실) | **버그 3·4와 결합** | `== → >=`만으로 부족(레벨 판정 전환 + TLAST 태깅 레이스). 설계 레벨 수정 묶음                                           |
| 3 (경계 오염)               | **버그 2·4와 결합** | 근본책은 flush 0-주입 제거(불필요한 주입이 오염 원인). 아래 "확인 필요 2" 참고                                             |
| 4 (홀수 parity)             | **버그 3과 결합**    | flush 제거 시 짝수 N은 위상 자동 보존 →`decimator_m2_phase0.v` 무수정 유지 가능. 홀수 길이 일반화 시에만 포트 추가 필요 |

버그 1만이 AXIS 래퍼 파일 내부의 진짜 국소 수정이다. 버그 2·3·4는 flush/종료판정/TLAST
태깅/데시메이터 위상이 얽힌 **하나의 설계 레벨 묶음**으로 봐야 한다. 다행히 근본책의
1순위 후보(flush 0-주입 제거, "확인 필요 2")가 성립하면 **`decimator_m2_phase0.v`
인터페이스 변경 없이** 짝수 N(데모는 항상 8192)에서 위상이 자동 보존되어, 이번 세션의
"v1 golden 무수정" 정책과의 충돌을 피할 수 있다. 다만 버그 2의 (b) TLAST 태깅 레이스는
flush 제거와 별개로 남으므로, 실제 착수 전 그 부분의 수정 설계는 별도 확인이 필요하다.

## 확인 필요 사항 (실제 착수 전)

1. **`scenario1_2.md`(100MHz PASS)가 정확히 몇 번째 UART 명령/캡처였는지 확인.**
   부팅 직후 첫 캡처였다면 버그 3의 오염을 피해 갔을 뿐이라는 설명이 성립한다. 만약
   여러 번째 캡처였다면 버그 3에 대한 위 가설이 틀렸다는 뜻이므로 원인을 다시 봐야 한다.
   - **결정적 실험**: FFT 기반 지표(SNR 등)는 앞쪽 stray 2비트 + 2샘플 시프트에
     둔감해서 오염돼도 PASS로 나올 수 있다. 같은 세션에서 **run 1과 run 2의 raw 캡처를
     샘플 단위로 diff**하는 게 하드웨어 확정 실험이다 — run 2 앞에 stray 2개 + 2샘플
     시프트가 보이면 버그 3이 실보드에서 재현됨을 확정.
2. **버그 3·4 근본책 — flush 0-주입 제거가 1순위 후보(신규).** 재검토 결과, flush 0-주입은
   파이프라인 드레인에 **원래 불필요**하다: `fir_n43`의 각 스테이지는 새 `in_valid`가 아니라
   *이전 스테이지의 valid*로 게이트되므로, 마지막 입력의 출력은 추가 입력 없이 L 사이클 뒤
   저절로 나온다(decimator도 단순 레지스터). 즉 flush 주입이 바로 stray 비트 / `out_cnt`
   잔류 / `keep_next` 위상 드리프트의 직접 원인이다. `core_in_valid`에서 flush 항을 빼고
   `waiting_for_last_out`은 새 패킷 입력 차단 용도로만 남기면:
   - 짝수 N(데모는 항상 8192)에서 `keep_next` 위상이 패킷 간 자동 보존 →
     **`decimator_m2_phase0.v` 무수정 유지 가능**("v1 golden 무수정" 충돌 해소).
   - 버그 4는 "홀수 패킷 미지원"이라는 제약 문서화로 축소되고, 위상 재동기화 포트 추가는
     홀수 길이까지 일반화할 때만 필요해진다.
   - 단, 버그 2의 (b) TLAST 태깅 레이스는 flush 제거와 별개로 남으므로 그 부분 수정 설계는
     따로 확정해야 한다.
3. **버그 1 수정 방식 결정 — 안 A(버퍼 확장) vs 안 B(게이트 일찍) 중 택1.**
   버퍼 깊이 실측·게이트별 비교·권장은 log 41에 정리됨(결론: 안 A). 착수 전 이 방식으로
   갈지만 확정하면 된다.
   - **향후(보류, 이번 범위 아님)**: 매직넘버 튜닝 대신 표준 AXI-Stream register-slice/FIFO로
     전환하면 latency 불변으로 correct-by-construction. 버그 2·3·4 재설계 때 함께 검토하는
     나중 과제로 남긴다.

## 전체 로드맵 — "FIR v2 이후" 다음 작업 계획

AXIS 버그 발견 이전에 세운 계획(`/plan` 요청 결과)을 여기 통합한다. AXIS 버그 조사로
순서가 재조정됐다 — **보드 실측은 AXIS 수정 완료 후로 순연**한다.

### Context

workflow_v21(v2 4-stage 파이프라인 + clk_wiz 정밀 스윕)로 Fmax를 v1 116MHz / v2 146MHz까지
확정했고, 안전마진을 둔 골든(v1 115MHz, v2 145MHz) 비트스트림도 재현 가능한 경로로 준비했다.
그다음 단계를 정리하던 중 AXIS 래퍼 버그를 발견해 순서가 바뀌었다.

핵심 전제 두 가지:

1. **CLAUDE.md의 "다음 작업 순서" 1~5번(FFT peak dB 정량화 등)은 이미 다 끝나 있다**
   (log 34, `docs/report/fir_n43/summary/scenario1_2.md`에 7/15/25/45MHz 톤별 수치·판정
   이미 존재). CLAUDE.md가 그 이후 안 갱신돼서 완료된 일이 "다음 작업"으로 남아 있었을
   뿐이다. → 재개할 일이 아니라 문서만 정정하면 된다.
2. **당초 남은 실질 작업은 보드 실측 하나였다** (workflow_v20의 "v1.0 완료 선언 조건" 5개
   중 마지막 "Fmax 정확도 검증"만 미완). 그런데 이제 그 실측이 AXIS 버그 수정 이후로
   순연됐다 — 버그 있는 래퍼로 먼저 재측정하면 결과 해석이 불가능하기 때문.

### 1. CLAUDE.md 문서 정정 (AXIS 버그와 무관, 언제든 진행 가능)

- "다음 작업 순서" 1~5번 항목 제거, 실제 결론 반영:
  - FFT peak dB 정량 출력 → 완료 (`fir_decimator_metrics.py`, log 34)
  - 7/15/25/45MHz 톤 판정 → 완료 (`scenario1_2.md` 표)
  - 25MHz/45MHz alias 문서화 → 완료 (같은 표에 포함)
  - plot x축 0~25MHz 수정 → **다른 방식으로 해결**: 크롭 대신 0~50MHz 유지 + invalid
    region(25~50MHz) 음영 처리로 결정됨 (`fir_decimator_fft_viewer.py`
    `OUTPUT_INVALID_REGION_MHZ`).
- "현재 기준 산출물" 섹션은 `docs/build_artifacts.md` 링크 추가 완료 상태 유지.
- 새 "다음 작업 순서"로 AXIS 버그 수정 → 보드 실측 순서를 명시.

### 2. AXIS 버그 수정 (이 문서 상단, 신규 최우선 순위)

위 "발견된 버그 4개" 절 참고. 버그 1(진짜 국소, skid 깊이 확장) 우선 → 버그 2·3·4
(flush 제거 + 종료판정 레벨화 + TLAST 태깅이 얽힌 하나의 설계 묶음, 1순위 근본책은
flush 0-주입 제거로 `decimator_m2_phase0.v` 무수정 유지 목표)는 범위 합의 후.

### 3. 보드 실측 절차 — v1@115MHz (workflow_v20 완결, AXIS 수정 이후 진행)

```bash
# 1. BOOT.bin 재생성 (v1 115MHz 골든 bit, AXIS 수정 반영된 비트스트림 사용)
vitis/fir_n43/rebuild_boot_image.sh \
  --bit build/fir_n43_v1_freq_115mhz/output/bd_fir_dma_wrapper.bit \
  --boot-out build/fir_n43_v1_freq_115mhz/output/BOOT.bin \
  --boot-tag FIR

# 2. SD 카드에 BOOT.bin 복사 → 보드 부팅 → UART에서 "READY FIR" 확인

# 3. 실측 실행 (버그 3 확인을 위해 동일 세션에서 2회 이상 명령 전송해볼 것)
uv run python sw/fir_decimator_report.py --mode 1-2 --port /dev/ttyUSB0
# → docs/report/fir_n43/summary/scenario1_2_115mhz.md 생성 확인

# 4. 결과 반영 (PASS 시)
#    - vivado/reports/sweep_summary.md correctness 컬럼 채움
#    - CLAUDE.md에 v1.0 완료 기록 (workflow_v20 "v1.0 완료 선언 조건" 5/5 충족)
#    - git commit
```

**FAIL 시 분기** (workflow_v20에 이미 기술됨): 110 MHz로 동일 절차 반복, Fmax(correctness)
110으로 정정.

### 4. 보드 실측 절차 — v2@145MHz (workflow_v21 마무리, AXIS 수정 이후 진행)

```bash
vitis/fir_n43/rebuild_boot_image.sh \
  --bit build/fir_n43_v2_freq_145mhz/output/bd_fir_dma_wrapper.bit \
  --boot-out build/fir_n43_v2_freq_145mhz/output/BOOT.bin \
  --boot-tag FIR_V2

uv run python sw/fir_decimator_report.py --mode 1-2 --port /dev/ttyUSB0
# → docs/report/fir_n43/summary/scenario1_2_v2_145mhz.md (신규 명명)
```

**결과 반영** (PASS 시):

- SNR/max_error가 v1 100MHz 수준(SNR 72.2dB, max_error 7 LSB)과 동일한지 확인 —
  크게 벗어나면 파이프라인 재배치(latency 3→4 cycle) 버그 의심 신호
- `vivado/reports/sweep_summary_v2.md`에 correctness 반영
- **v1↔v2 교체 여부는 이 결과가 나온 뒤에 결정** (사용자 확인됨 — 지금 미리 정하지 않음)

**⚠ 확인 필요한 리스크**: `rebuild_boot_image.sh`는 "Vivado 하드웨어나 Vitis 플랫폼은
재생성하지 않는다"고 명시되어 있다 — 기존 `build/fir_n43/vitis` 워크스페이스(플랫폼은
v1 100MHz 기준으로 export됨)를 그대로 재사용하며 비트스트림만 바꾼다. v2는 top 모듈이
다르고 clk_wiz IP가 추가됐지만 PS7 설정(DDR 등)과 AXI-Stream 포트 구성 자체는 동일하므로
이론상 문제없어야 하지만, 실제로 안 해본 조합이다. 첫 시도에서 fsbl/플랫폼 불일치로 안
되면 v2 XSA로 Vitis 플랫폼을 별도로 새로 export해야 할 수 있다.
