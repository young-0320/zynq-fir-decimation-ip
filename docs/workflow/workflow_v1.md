# FIR Decimation 프로젝트 워크플로우 v1

- 작성일: 2026-03-08
- 목적: ideal Python reference model 단계까지를 재현 가능하게 운영

## 1) 완료 조건 (Definition of Done)

1. `model/ideal` 참조 모델 동작 검증 완료(FIR -> decimation 체인).
2. ideal 단계 테스트가 `sim/python/test/ideal` 기준으로 통과함.
3. ideal 단계 구현/테스트 결과와 재현 명령어가 문서에 남아 있음.
4. 이후 입력 신호 확정 단계(v2)로 넘어가기 위한 스펙/로그/기준 모델이 정리되어 있음.

## 2) 주간 루틴

1. 월-수: 기능 구현 + 단위 테스트(TDD).
2. 목: 인터페이스/예외/출력 길이 등 구현 계약 점검.
3. 금: 로그 정리(`docs/log/*.md`) + 다음 주 TODO 확정.
4. 주말: Velog 초안 업데이트(결정-근거-결과 구조).

## 3) 하루 운영 규칙

1. 하루 마감 10분 동안 아래 5줄을 `docs/log`에 전사:
   - 오늘 결정 1개
   - 근거 수치 1개
   - 실패/수정 1개
   - 재현 명령어 1줄
   - 다음 액션 1개

## 4) 산출물 위치 규칙

1. 스펙/의사결정: `docs/spec/ideal_model_spec.md`, `docs/log/*.md`
2. 이상 모델 코드: `model/ideal/*.py`
3. 테스트: `sim/python/test/ideal/*.py`
4. 실험 보조 스크립트: `sim/python/*.py`
5. 이미지/그래프: `docs/asset/*`

## 5) 의사결정 기록 템플릿

1. 문제 정의
2. 제약 조건(시간/자원/스펙)
3. 대안(A/B/C)
4. 선택안
5. 근거 데이터(숫자/리포트)
6. 트레이드오프와 후속 계획

## 6) 품질 게이트 (각 단계 공통)

1. 코드 실행 가능 (`uv run` 기준)
2. 입력 검증/예외 처리 존재
3. 테스트 최소 1개 이상 추가
4. 결과 수치가 문서에 기록됨
5. 재현 명령어가 문서에 존재함

## 7) 업데이트 정책

1. `workflow_v1.md`는 주 1회만 수정(운영 흔들림 방지).
2. 실험/결정 기록은 `docs/log`에 매일 누적.
3. v1의 스코프는 ideal Python model 단계까지로 한정한다.
4. 입력 신호 확정 단계부터는 `workflow_v2.md`를 기준 문서로 사용한다.
