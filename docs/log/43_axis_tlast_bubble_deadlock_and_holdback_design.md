## 배경

log 42가 구현한 최소 패치(`==→>=`, skid 3→4칸, flush 0-주입 제거, `| s_axis_tlast` tready 추가)를
코드 리뷰(`/code-review high`)로 검토하는 과정에서, **workflow_v22 버그2 note (a)가 이미 예견했던
"레벨 판정 전환 없이는 여전히 잠긴다"는 우려가 실제로 남아있음**을 directed 시뮬레이션으로
확인했다. log 42 §2가 고친 것은 "backpressure로 인한 지연" 경로였고, **"마스터 tvalid 버블로
인한 지연" 경로는 안 고쳐져 있었다.**

---

## 1. 재현 (확정)

증상: 다운스트림 backpressure가 전혀 없어도(`m_axis_tready=1` 고정), **s_axis_tlast 비트
직전에 tvalid 버블이 일정 사이클 이상**이면 TLAST가 영구 유실되고 래퍼가 리셋 전까지
영구 데드락한다.

- v1(코어 latency 4 cycle): 버블 ≥ 3사이클에서 재현
- v2(코어 latency 5 cycle): 버블 ≥ 4사이클에서 재현

재현 TB(리뷰 세션 스크래치패드, repo 미편입): 버블 0~6 스윕, `m_axis_tready=1` 고정,
N_IN=8 소패킷. v1은 버블 3부터, v2는 버블 4부터 전부 FAIL(수신 개수는 맞으나 TLAST=0).

**근본 원인**: `target_out_cnt`/`waiting_for_last_out`은 tlast 입력이 **수락되는 순간**
확정된다. 그런데 코어의 실제 마지막 출력(짝수 N 가정에서 입력 N-2, keep 위상)은
그보다 먼저 나올 수 있다 — 입력 N-2와 입력 N-1(tlast)의 수락 시점 사이에 코어 latency(v1=4/v2=5)
이상의 간격(버블)이 있으면, 마지막 출력이 `waiting_for_last_out=0`인 채로 태그 없이
버퍼에 실린다. 이후 `waiting_for_last_out`이 1이 될 때는 `out_cnt`가 이미 target을
지나쳐 있고, 그 값을 다시 평가할 core_out_valid 펄스가 안 온다 → 영구 잠김.

**왜 log 42가 못 잡았나**: log 42의 `tb_skid_stress.sv`는 zero-bubble 연속 입력만 검증한다
(파일 자체 주석에도 "continuous zero-bubble input"으로 명시). 골든 TB S2(`tb_fir_decimator_n43_axis[_v2].sv`)는
`drive_packet(3)`로 버블을 최대 3사이클만 주므로, v1 기준 **경계값과 정확히 같아서
현재 시드로는 우연히 안 걸렸을 뿐** — 시드가 바뀌면 `run_all`이 랜덤하게 FAIL할 수 있는
상태였다.

**실사용 연관**: 이 경로가 실보드에서 배제되지 않는 이유 — `vivado/fir_n43/bd_fir_dma.tcl`에서
`axi_dma_0`의 MM2S/S2MM이 `axi_mem_intercon`을 거쳐 **같은 HP0 포트**로 DDR에 접근한다
(`bd_fir_dma.tcl:623-624`). S2MM 쓰기와 MM2S 읽기가 경합하면 MM2S 쪽 버스트에 스톨이
생겨 스트림 tvalid 갭이 발생할 수 있다 — "마스터가 버블을 안 만든다"고 가정할 근거가 없다.

## 2. 수정 설계 결정 (확정)

세 가지 수준을 검토했다:

| 수준 | 내용 | 남는 구멍 |
|---|---|---|
| (a) 잠김만 해제 | 종료 판정을 pulse-gated에서 매 사이클 레벨 체크로 전환 | FSM은 풀리지만 그 패킷의 TLAST는 이미 유실 — 여전히 1패킷 실패 |
| (b) (a) + 소급 태깅 시도 | 버퍼에 아직 남아있으면 뒤늦게 태그 | 버블이 길어 이미 트럭(m_axis)에 실려 나갔으면 복구 불가 |
| (c) (b) + 출력 보류(hold-back) | "마지막일 수도 있는" 최신 출력을 확정 전까지 m_axis에 노출 안 함 | 없음(아래 근거) |

**결정: (c) hold-back 방식 채택.** (a)/(b)는 "고쳤다"고 선언할 수 없는 잔여 실패 케이스가
남는다. (c)는 래퍼 내부만으로 완결되어 "코어 무수정" 원칙(workflow_v22 확정 결정)을
그대로 유지한다.

### (c)의 구체 메커니즘

- 코어가 출력을 낼 때마다 **가장 최근 출력 하나만** "미확정(마지막일 수도 있음)"으로 표시.
- 그다음 출력이 나오면 → 직전 것은 "마지막 아님" 확정 → 정상 태그(0)로 드레인 허용.
- s_axis_tlast가 **수락**되는 순간 → 그 시점의 `out_cnt`(이미 신뢰 가능, 무조건 카운트됨)와
  방금 확정될 `target_out_cnt`를 비교:
  - `out_cnt == target` → 미확정 슬롯이 바로 그 마지막 출력 → 그 슬롯의 tlast 비트를
    소급 설정, 드레인 허용.
  - `out_cnt < target` → 미확정 슬롯은 마지막이 아님(더 나올 예정) → 드레인 허용(태그 0),
    실제 마지막은 이후 기존 pulse-gated 로직(`waiting_for_last_out && out_cnt+1>=target`)이
    정상 처리(이 경로는 버블이 짧은 정상 케이스와 동일 — 미변경).
- **핵심 규칙**: 미확정 슬롯은 확정되기 전까지 절대 `m_axis`로 전달되지 않는다(선반 맨 앞에
  도달해도 그 자리에서 대기). 구현상 `m_axis_tvalid`/`transfer`에 미확정 플래그의 부정을
  추가 게이팅.

### 버퍼 깊이(4칸)에 대한 영향 — 사전 분석

미확정 상태로 드레인이 막히는 구간은 **오직 "tlast 직전 버블"이 실제로 진행 중인 동안**만
발생하고, 그 정의상 그 구간에는 새 입력이 전혀 안 들어온다(마스터가 버블 중이므로).
따라서 새 완성품이 그 정체 구간 동안 추가로 쌓이지 않아 depth=4 사이징이 깨지지 않는다는
게 1차 분석 결론이다. **단, 이는 분석이며 실제 결정 근거는 §4의 재검증 실험이다** —
`tb_buf_depth` 류 점유량 실측을 hold-back 로직 반영 후 다시 돌려 확인한다.

### 종료 판정 레벨화 (병행 수정)

hold-back과 별개로, `waiting_for_last_out` 해제 조건을 `core_out_valid` 펄스 게이트에서
매 사이클 레벨 체크로 전환한다(`if (waiting_for_last_out && out_cnt >= target_out_cnt)`를
`always` 블록에서 pulse 조건 밖으로). hold-back만으로는 TLAST 태깅은 해결되지만, 코어가
완전히 드레인되어 더 이상 core_out_valid 펄스가 안 오는 극단적 케이스에서 FSM 자체가
안 풀리는 잔여 가능성을 막기 위한 방어.

   여전히 최대 점유량 상한임을 재확인 — 안 맞으면 깊이 재산정.
4. 코어(`fir_n43[_v2].v`, `decimator_m2_phase0.v`)·BD/Tcl 무수정 유지.
